# Guided Work Mode — рабочий процесс менеджера и founder

Этот документ описывает связку экранов, которые превращают приоритизированную
очередь лидов в пошаговый дневной рабочий процесс — отдельно для менеджера по
продажам и отдельно для founder/владельца агентства.

## Для менеджера: дневной цикл

```
/workday  →  /workday/focus/[id]  (повторяется для каждого лида)  →  /workday/summary
```

### `/workday` — "Мой день" (Manager Today Queue)

Главный экран дня менеджера. Показывает:

- Метрики дня: горячих лидов, просрочено по SLA, денежный пайплайн
  (сумма Money Priority Score по всем лидам).
- Единую очередь лидов, отсортированную по `urgencyRank`:

  ```
  rank = (slaStatus === "overdue" ? 1000 : slaStatus === "warning" ? 500 : 0)
       + (temperature === "hot" ? 300 : temperature === "warming" ? 150 : 0)
       + moneyPriorityScore
  ```

  То есть приоритет сортировки буквально: **просрочка по SLA → горячая температура
  → денежный приоритет**, именно в этом порядке весов (1000 > 500 > 300 > 150,
  Money Score складывается поверх как тонкая настройка внутри одного уровня
  срочности).

- Для каждого лида — рекомендованное следующее действие (`nextBestAction`),
  которое явно называет причину срочности ("Срочно: ... SLA уже просрочен" /
  "лид горячий, не дать остыть" / "дедлайн SLA приближается").
- Кнопка "Начать →" на каждой строке ведёт в Focus Mode для этого лида.

### `/workday/focus/[id]` — Focus Mode

Экран "один лид за раз" — менеджер открывает конкретный лид из очереди и работает
с ним без распыления внимания на остальную очередь: видит контекст лида, скрипт
для текущей ситуации (из `/scripts`), готовое сообщение для отправки, и совершает
действие (написать, позвонить, отправить КП), после чего система переходит к
следующему лиду в очереди (`app/api/workday/focus-next/route.ts` уже спроектирован
под эту логику "что дальше после этого лида").

### `/workday/summary` — End of Day Summary

Итог дня: сколько лидов обработано, сколько задач закрыто, что осталось на завтра.
Завершает цикл, давая менеджеру ощущение прогресса и фиксируя данные для
`manager_coaching_stats` (лидерборд на `/founder`).

### Вспомогательные экраны менеджера

- **`/quick-wins`** — отдельная подвыборка лидов, которые можно закрыть быстро
  с минимальным усилием (например, высокая Deal Probability при низком пороге
  усилий) — способ дать менеджеру лёгкие победы между сложными лидами и
  поддержать мотивацию/темп в течение дня.
- **`/scripts`** — библиотека готовых речевых модулей (`SalesScript`), организованных
  по `triggerType` и обработке возражений (`objectionHandled`) — менеджер открывает
  нужный скрипт по ситуации прямо во время разговора/переписки.
- **`/replies`** — лента входящих ответов клиентов с классификацией
  (`ReplyClassificationLabel`: `interested`, `objection_price`, `not_interested` и т.д.)
  и предложенным следующим действием (`suggestedNextAction`) — менеджер быстро
  видит, что ответил клиент, и что с этим делать дальше, без необходимости
  перечитывать всю переписку с нуля.

### Итоговый дневной поток для менеджера

1. Зайти в `/workday`, увидеть приоритизированную очередь на сегодня.
2. Открыть верхний лид через Focus Mode (`/workday/focus/[id]`).
3. Прочитать контекст, использовать подходящий скрипт (`/scripts`), отправить
   сообщение/совершить звонок.
4. Перейти к следующему лиду (или заглянуть в `/quick-wins` для лёгкой победы).
5. Между делом проверять `/replies` на новые ответы клиентов.
6. В конце дня — `/workday/summary`, чтобы зафиксировать результат и увидеть,
   что осталось на завтра.

## Для founder/владельца: `/founder` (Founder Control Center)

Это отдельный обзорный экран, не для повседневной обработки лидов, а для
**контроля бизнеса целиком**:

- Метрики верхнего уровня: горячих лидов сегодня, лидов под риском SLA,
  денежный пайплайн (сумма Money Score), число лидов с активным Founder Boost.
- **Лидерборд менеджеров** (`managerCoachingStats`): среднее время ответа,
  конверсия, число обработанных лидов, отклонение от средней по команде —
  инструмент коучинга и распределения нагрузки.
- **Топ лиды по денежному приоритету** — те же Money Score-based лиды, что в
  очереди менеджера, но агрегированные на уровне всей компании, а не одного
  менеджера, со ссылками на карточку лида.
- **Founder Boost** — механизм, позволяющий founder вручную поднять приоритет
  конкретного лида (`isFounderBoosted` в `lib/types.ts`/`lib/mock.ts`,
  `app/api/founder/boost/route.ts`), когда founder видит стратегическую
  важность лида, которую формулы не улавливают (например, известный клиент,
  PR-ценность сделки) — единственная точка ручного вмешательства в
  автоматическую приоритизацию.

### Использование founder

Founder открывает `/founder` не для обработки конкретных лидов (для этого есть
менеджеры и `/workday`), а чтобы:

1. За секунды понять состояние пайплайна целиком (горячие лиды, риски SLA, деньги).
2. Оценить, кто из менеджеров отрабатывает лучше/хуже команды — для коучинга
   или перераспределения нагрузки.
3. Точечно "форсировать" 1-2 стратегически важных лида через Founder Boost,
   не трогая остальную логику приоритизации.

## Связанные, но отдельные от Guided Work Mode экраны

- **`/offers`** (Offer Testing Lab) — управление тестируемыми пакетами
  размещения (`Offer`), не входит в ежедневный workflow напрямую, но
  питает рекомендации (`offer_recommendation` в `AI_LAYER.md`) внутри Focus Mode.
- **`/sources`** — мониторинг состояния источников сигналов (расширенная версия
  существующего демо-экрана), используется скорее администратором/founder, чем
  менеджером в рамках ежедневной работы.
- **`/leads/[id]`** — детальная карточка лида (расширена в этом раунде), к
  которой Focus Mode и остальные экраны линкуются, но которая может открываться
  и вне Guided Work Mode (например, из поиска или истории сделок).
