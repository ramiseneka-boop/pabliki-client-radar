# Источники сигналов (Source Connectors)

Реестр всех источников сигналов живёт в `lib/sources.ts` и типизирован через
`SourceConnector` в `lib/types.ts`. Ниже — полная таблица того, что там сейчас
реально записано (без выдуманных значений), плюс заметки о том, что нужно для
реальной реализации каждого источника.

Статусы (`status`) в `lib/sources.ts`:
- **`active`** — источник принимает данные прямо сейчас (но это всё ещё ручной/демо-поток).
- **`mock`** — данные показываются, но они статичны/придуманы, реального сбора нет.
- **`pending_access`** — функциональность спроектирована, но блокирована отсутствием
  ключа/токена/доступа.

и `lib/scoring.ts` для взвешивания, насколько доверять сигналу от конкретного

"активны" в том смысле, что их структура не блокирована ничем внешним — их реализация
в Фазе 1+ это вопрос написания парсера/приёмника, а не получения доступа.

## Заметки по каждому источнику для реальной реализации

### `pabliki_site` — Сайт Pabliki.kz (webhooks)
Вебхук-приёмник заявок с собственного сайта агентства. Не требует внешних ключей —
это **самый дешёвый источник для реализации**: нужен только реальный HTTP-эндпоинт
(уже есть форма заглушки в `app/api/signals/ingest/route.ts`), который сайт будет
вызывать при отправке формы. Юридически — это собственные данные клиентов, которые
сами оставили заявку (явное согласие), наименее рискованный с точки зрения
персональных данных источник.

### `google_places` — Google Places
Требует `GOOGLE_PLACES_API_KEY` из Google Cloud Console. У Places API есть
бесплатный месячный кредит, после которого — оплата по использованию (Nearby
Search / Place Details эндпоинты). Хорошо подходит для отслеживания "новых
филиалов/точек" по категории и городу. Заглушка уже существует:
`app/api/connectors/google-places/route.ts`.

### `2gis` — 2GIS
Требует `TWOGIS_API_KEY`. **У 2GIS нет публичного прайс-листа на коммерческий
доступ к API** — условия и стоимость нужно уточнять напрямую через партнёрский
портал 2GIS. Это не блокер, но означает, что бюджетирование этого источника
нельзя сделать заранее без обращения к 2GIS. Заглушка: `app/api/connectors/2gis/route.ts`.

### `rss` — Новости / RSS
Не требует ключей вообще — RSS-ленты публичны. Реализация = добавить парсер RSS
(например `rss-parser` npm-пакет) и список лент по нишам/регионам. Хороший
кандидат для раннего внедрения наравне с `pabliki_site`.

### `wordstat` — Поисковые тренды / Wordstat
Yandex Wordstat не имеет официального публичного API — практические интеграции
обычно идут через сторонние обёртки или скрейпинг интерфейса, что менее стабильно
и может нарушать условия использования Яндекса. Перед реализацией стоит явно
оценить юридический риск и стабильность подхода (см. `IMPLEMENTATION_ROADMAP.md`
Фаза 0).

### `meta_ads` — Meta Ad Library
Требует `META_ACCESS_TOKEN` и `META_AD_ACCOUNT_ID`. Сам Ad Library API бесплатен
для чтения публичных объявлений, но имеет **строгие rate limits** и требует
прохождения Meta App Review для production-уровня доступа — это организационная
задержка (недели), а не только техническая. Сигнал "у компании активная реклама" —
сильный индикатор бюджета, отсюда высокий `nicheMoneyWeight`/`reliability` (0.7).

### `instagram` — Instagram (бизнес-аккаунты)
Требует `INSTAGRAM_ACCESS_TOKEN`. Instagram Graph API даёт доступ преимущественно
к собственным/связанным бизнес-аккаунтам, а не произвольным сторонним — реальный
охват источника может оказаться значительно у́же, чем "мониторинг любых бизнес-аккаунтов
в нише". Перед инвестированием в реализацию стоит явно проверить, какие данные о
*сторонних* аккаунтах API в принципе отдаёт легально.

### `telegram` — Telegram-каналы
Требует `TELEGRAM_BOT_TOKEN`. Telegram Bot API бесплатен, но боты по умолчанию не
могут читать сообщения каналов, в которые их не добавили как админа — реализация
этого источника на практике означает мониторинг конкретного списка публичных
каналов, куда бот добавлен, а не "все Telegram-каналы в нише". Тот же токен
пригодится и для исходящих уведомлений менеджерам (Фаза 2 в роадмапе).

### `goszakup` — Госзакупки KZ (goszakup.gov.kz)
Требует `GOSZAKUP_API_TOKEN`. Это **публичный государственный API** — относительно
недорогой/бесплатный путь получения сигналов о тендерах на маркетинговые/рекламные
услуги, но с регламентированными лимитами обращений на стороне госпортала.
Регистрация разработчика происходит через сам портал goszakup.gov.kz.
Хорошее соотношение ценность/стоимость для Фазы 3.

### `csv_import` / `old_clients_import` — Excel/CSV импорт
Не требует ключей. Реализация = парсер файлов + дедупликация (таблицы
`import_batches`/`import_rows`/`dedup_candidates` уже в `db/schema.sql`).
Хороший способ быстро наполнить БД историческими данными агентства без ожидания
внешних API.

### `manual` — Ручное добавление
Уже `status: "active"` — единственный источник без статуса "mock"/"pending_access",
потому что это просто форма для менеджера, не требующая внешней интеграции вообще.

### `hiring_signals`, `events`, `reviews` — вторичные сигналы
Все три — `mock`, без `required_env_vars`. Это сигналы, которые потенциально можно
собирать через комбинацию RSS/скрейпинга/партнёрских API (вакансии — HH.kz,
мероприятия — городские афиши, отзывы — Google/2GIS отзывы). Их реализация
скорее про инженерные усилия (парсинг), чем про получение доступа — хорошие
кандидаты для постепенного добавления после Фазы 1.

## Закон о персональных данных РК — общая заметка

Закон Республики Казахстан "О персональных данных и их защите" требует
**локализации баз персональных данных граждан РК** — то есть физического хранения
на серверах в Казахстане для определённых категорий данных. Это относится прежде
всего к данным физических лиц (контакты ЛПР, телефоны, переписка), но не
обязательно к собственно бизнес-данным компаний (БИН, название, адрес). Перед
включением Фазы 1 (выбор региона Supabase) и Фазы 2 (хранение переписки) стоит
явно зафиксировать, какие поля считаются персональными данными физлица и
требуют локального хранения — это решение находится вне кода и должно быть
принято владельцем продукта/юристом, а не подразумеваться по умолчанию.
