# AI-слой: текущее состояние и план внедрения

## 1. Где AI сейчас (короткий ответ: нигде)

В прототипе **нет ни одного реального вызова к LLM**. Всё, что выглядит как
"AI-сгенерированный" текст — это статичные строки, написанные вручную и
захардкоженные в `lib/mock.ts`:

- `whatsappMessage`, `followUp1d`, `followUp3d` — поля у каждого mock-лида,
  например: `"Здравствуйте! Заметили старт продаж ЖК Prime House в Астане —
  поздравляем с запуском. Под старт продаж хорошо отрабатывают городские,
  районные и новостные паблики Астаны…"` — это написанный человеком текст,
  не результат генерации.
- `scriptText` в массиве скриптов продаж (`/scripts`, `app/api/scripts/route.ts`) —
  те же статичные строки на русском, описывающие, что сказать менеджеру в той или
  иной ситуации (возражение по цене, запрос на отправку КП и т.д.).
- Классификация ответов (`app/api/leads/[id]/classify-reply/route.ts`,
  тип `ReplyClassification` в `lib/types.ts`) — в mock-данных классификация уже
  "проставлена" заранее, никакой реальной NLP-логики нет.

API-маршрут `app/api/leads/[id]/messages/route.ts` явно документирует это в
своём `meta.note`:

> "Тексты сообщений — заранее написанные mock-строки из lib/mock.ts. В проде
> это будет AI-генерация (OpenAI/Anthropic) по шаблонам message_templates +
> персонализация по lead/company."

`.env.example` уже резервирует переменные `OPENAI_API_KEY` и `ANTHROPIC_API_KEY`
под будущую реализацию — но обе переменные сейчас не используются ни в одном
файле кода (только проверяются на "задано ли" в `app/api/health/route.ts`,
чтобы показать инфраструктурный статус `ai_messages` как "не настроено").

## 2. Какие mock-фичи становятся реальными LLM-вызовами

| Текущая mock-фича | Файл сейчас | Что заменится на реальный AI-вызов |
|---|---|---|
| `whatsappMessage` / `followUp1d` / `followUp3d` | `lib/mock.ts`, отдаётся через `app/api/leads/[id]/messages/route.ts` | Генерация персонализированного первого сообщения и follow-up'ов по лиду |
| Классификация ответа клиента | `app/api/leads/[id]/classify-reply/route.ts` | Реальная классификация текста ответа в одну из категорий `ReplyClassificationLabel` |
| Обогащение лида (резюме компании) | `app/api/leads/[id]/enrich/route.ts` | Сжатие сырых данных о компании (сайт, отзывы, соцсети) в короткое резюме для менеджера |
| `scriptText` в скриптах продаж | `lib/mock.ts`, `app/api/scripts/route.ts` | Персонализация общего скрипта под конкретную нишу/ситуацию лида |
| Объяснение приоритета (`score_explanation`) | поле есть в `db/schema.sql` (`leads.score_explanation`), в коде не заполняется | Текстовое объяснение, почему лид получил такой Priority/Money Score — числа считает детерминированный код (`lib/scoring.ts`), AI только формулирует объяснение человеку |
| Сводка для founder (`/founder`) | `app/api/founder/stats/route.ts` (статичная агрегация) | Короткий ежедневный текстовый брифинг поверх тех же агрегированных цифр |

Важный принцип, уже заложенный в комментарии `lib/scoring.ts`: **"Числа считает код;
AI лишь формулирует объяснение."** AI-слой добавляется как слой *интерпретации и
генерации текста* поверх детерминированных вычислений — не заменяет scoring/temperature/
SLA/money-engine логику.

## 3. Предлагаемые категории промпт-шаблонов

Ниже — предлагаемый набор из 15 категорий промптов для библиотеки шаблонов
(концептуально соответствует таблице `message_templates` в `db/schema.sql`, которая
сейчас покрывает только `channel`/`template_type` для исходящих сообщений — для
остальных категорий понадобится либо расширить эту таблицу, либо завести отдельную
таблицу `ai_prompt_templates`).

1. **`first_contact_message`** — первое сообщение лиду по триггеру (аналог текущего `whatsappMessage`).
2. **`follow_up_1d`** — follow-up через 1 день без ответа.
3. **`follow_up_3d`** — follow-up через 3 дня без ответа.
4. **`objection_price_response`** — ответ на возражение "дорого".
5. **`objection_send_proposal`** — ответ на просьбу "пришлите КП".
6. **`reply_classification`** — классификация входящего ответа клиента в `ReplyClassificationLabel`.
7. **`lead_summary_for_manager`** — короткое резюме лида перед звонком/перепиской (для Focus Mode `/workday/focus/[id]`).
8. **`next_best_action_explanation`** — объяснение, почему именно это действие сейчас приоритетно.
9. **`deal_risk_flag`** — флаг риска "сделка может развалиться" с объяснением причины.
10. **`lost_reason_inference`** — предположение причины проигрыша сделки на основе переписки (входит в `LostReason`).
11. **`offer_recommendation`** — рекомендация подходящего оффера из Offer Testing Lab под нишу/бюджет лида.
12. **`script_personalization`** — персонализация общего `SalesScript` под конкретного лида.
13. **`enrichment_company_summary`** — резюме компании по сырым данным обогащения (сайт, соцсети, отзывы).
14. **`sla_escalation_message`** — текст уведомления менеджеру/founder при приближении/нарушении SLA.
15. **`founder_daily_briefing`** — ежедневный текстовый брифинг для Founder Control Center.

Дополнительные кандидаты сверх обязательного минимума (по необходимости):

16. **`competitor_mention_alert`** — реакция на упоминание конкурента в переписке.
17. **`reactivation_message`** — сообщение для реактивации старого клиента (`old_clients_import`).
18. **`tender_summary`** — резюме тендера с goszakup.gov.kz в формате, понятном менеджеру.

## 4. Технические требования к реализации

- **Provider abstraction.** Поддержать оба ключа (`ANTHROPIC_API_KEY`, `OPENAI_API_KEY`)
  через общий интерфейс — не привязываться к одному провайдеру в коде маршрутов.
- **Graceful fallback — обязательное поведение.** Если ни один ключ не задан (как
  сейчас, по умолчанию), API-маршрут должен **продолжать отдавать текущие mock-строки
  из `lib/mock.ts`** с `meta.status: "mock"`, а не возвращать ошибку. Псевдологика:

  ```
  if (ANTHROPIC_API_KEY || OPENAI_API_KEY) {
    // реальный вызов LLM по шаблону, meta.status = "active" (или убрать meta вовсе)
  } else {
    // вернуть существующее mock-поле, meta.status = "mock", как сейчас
  }
  ```

  Это тот же паттерн, что уже используется для источников сигналов в
  `lib/sources.ts`/коннекторах — единообразие важно для предсказуемости API.
- **Логирование результатов.** Для категорий 6 (классификация) и 9-10 (риск/причина
  проигрыша) стоит логировать AI-вывод отдельно от детерминированных полей, чтобы
  потом можно было оценить точность и дообучить промпты — таблица
  `reply_classifications.classified_by` в `db/schema.sql` уже предусматривает
  значение `'ai_model'` в дополнение к `'heuristic'`/`'manager'`.
- **Cost control.** Для высокочастотных категорий (1-3, генерация сообщений)
  стоит заранее оценить объём вызовов в день × стоимость токенов — это
  единственная статья расходов в AI-слое, которая растёт линейно с объёмом лидов.

## 5. Что НЕ нужно делать

Не нужно подключать AI для самого Priority/Temperature/SLA/Money Score —
эти движки детерминированные и явно спроектированы так, чтобы оставаться
объяснимыми и воспроизводимыми (см. `SCORING_ENGINE.md`). AI отвечает только за
текст, человеческую интерпретацию и неструктурированный анализ (классификация,
резюме), а не за сами числа приоритета.
