# Pabliki Client Radar — архитектура (текущая и целевая)

Этот документ объясняет, как сейчас устроена система, какой путь до production-версии,
и где проходит граница между «реальным кодом» и «mock-данными/заглушками».

## 1. Текущее состояние (как есть сегодня)

Pabliki Client Radar — это **монолит на Next.js (App Router)**, который выглядит и
ведёт себя как production-приложение, но **полностью работает на статичных
mock-данных в памяти процесса**. Никаких внешних сетевых вызовов, никакой базы
данных, никакой авторизации.

Ключевые факты:

- **Frontend + backend в одном приложении.** Next.js App Router рендерит страницы
  (`app/**/page.tsx`) и одновременно отдаёт ~39 mock API-маршрутов (`app/api/**/route.ts`),
  которые UI вызывает так, как если бы это был отдельный backend.
- **Источник данных — `lib/mock.ts`.** Все лиды, офферы, скрипты, задачи и т.п. — это
  TypeScript-массивы и объекты, захардкоженные в файле и пересоздаваемые при каждом
  запуске процесса. Никакой персистентности нет: перезапуск сервера = чистый слайс
  тех же mock-данных (они статичны, поэтому "сброса" в привычном смысле не происходит,
  но никакая правка через UI не сохраняется в файл).
- **Нет базы данных.** `db/schema.sql` — это **черновик-артефакт**, 57 таблиц, описывающих,
  как *могла бы* выглядеть схема Postgres/Supabase. Файл не применяется ни к какой БД,
  не подключён к ORM/миграционному инструменту и не исполняется приложением.
- **Нет авторизации.** Нет логина, нет ролей, нет сессий. Типы `users`/`user_role` есть
  только в черновой схеме БД.
- **Нет внешних интеграций.** Ни одного реального HTTP-вызова к Google Places, 2GIS,
  Meta, WhatsApp Business API, Telegram, goszakup.gov.kz, OpenAI или Anthropic.
  `.env.example` перечисляет переменные для всех этих интеграций — и все они пустые.
- **Деплой.** Статический Next.js-проект, который можно задеплоить на Vercel
  без каких-либо secrets — приложение полностью самодостаточно на mock-данных.

### Что это означает практически

Каждый mock API-маршрут возвращает объект с полем `meta: { status, note }`, где
`status` — это `"mock"` (данные есть, но придуманные/статичные) или
`"pending_access"` (функциональность спроектирована, но реализация блокирована
отсутствующим внешним доступом/ключом). Это сделано осознанно — чтобы при переходе
к реальным интеграциям было видно, какие маршруты нужно "включить", не меняя
контракт API.

## 2. Целевая production-архитектура

```
                     ┌─────────────────────────────┐
                     │   Next.js (App Router)      │
                     │   frontend + API routes     │
                     └─────────────┬────────────────┘
                                   │
                 ┌─────────────────┼──────────────────┐
                 │                 │                  │
                 ▼                 ▼                  ▼
        ┌────────────────┐ ┌──────────────┐  ┌──────────────────┐
        │ Supabase        │ │ Background    │  │ AI-слой           │
        │ Postgres        │ │ jobs / cron   │  │ (Anthropic/OpenAI)│
        │ (db/schema.sql) │ │ (опрос        │  │ генерация         │
        │ + Supabase Auth │ │ источников)   │  │ сообщений,        │
        └────────────────┘ └──────┬───────┘  │ классификация     │
                                   │           └──────────────────┘
                                   ▼
                     ┌─────────────────────────────┐
                     │  Внешние источники сигналов  │
                     │  Google Places, 2GIS, Meta,  │
                     │  Telegram, goszakup.gov.kz…  │
                     └─────────────────────────────┘
```

Архитектурно ничего революционного не требуется: тот же Next.js монолит сохраняется
как frontend и API-слой, но три вещи появляются "под капотом":

1. **Реальная БД** (Supabase Postgres) вместо `lib/mock.ts` — API-маршруты начинают
   делать `SELECT`/`INSERT` вместо `find()` по массиву.
2. **Фоновые задачи** (cron / scheduled functions / отдельный worker) — опрос внешних
   источников по расписанию (`lib/sources.ts.next_run_at` уже моделирует это поле).
3. **AI-слой** — вызовы LLM для генерации текстов и классификации, с graceful fallback
   на mock-тексты, если ключи не настроены (см. `AI_LAYER.md`).

## 3. Сквозной поток данных (signal → revenue)

```
[Источники сигналов]              lib/sources.ts (коннекторы)
   Google Places, 2GIS, Meta Ads,
   RSS, Wordstat, goszakup, ручной
   ввод, CSV-импорт, вебхуки сайта
        │
        ▼
[Сигнал / Signal]                 app/api/signals/ingest
   raw_payload → нормализация в
   Lead-кандидата
        │
        ▼
[Scoring / Temperature / SLA / Money Engine]
   lib/scoring.ts      → Priority Score (1..100, "насколько лид качественный")
   lib/temperature.ts  → hot/warming/cooling/cold/expired (time-decay по триггеру)
   lib/sla.ts          → on_time/warning/overdue (дедлайн первого контакта)
   lib/moneyScore.ts   → Money Priority Score (сколько денег под риском и как срочно)
   lib/validation.ts   → Minimum Viable Lead? Source Confidence?
        │
        ▼
[Lead]                             app/api/leads, app/leads/[id]
   попадает в очередь менеджера, если прошёл валидацию
        │
        ▼
[Guided Work Mode]                /workday → /workday/focus/[id]
   менеджер обрабатывает лид по приоритету
   (overdue SLA → hot temperature → money score)
        │
        ▼
[Manager Action]                  написать WhatsApp / позвонить / отправить КП
   app/api/leads/[id]/messages, app/api/scripts, app/api/replies
        │
        ▼
[Outcome / Revenue Attribution]   deals, deal_stage_history, revenue_attribution
   (в схеме БД — пока не реализовано в коде)
   app/founder, app/api/analytics/revenue-attribution
```

Узлы "Scoring Engine", "Lead", "Guided Work Mode" — это **реальный исполняемый
TypeScript-код**, работающий прямо сейчас на mock-входных данных. Узлы "Источники
сигналов" (кроме `manual`/`csv_import`/`pabliki_site`, которые тоже mock) и "Outcome /
Revenue Attribution" — это пока только спроектированный контракт (типы + API-заглушки
+ черновая схема таблиц), без реальной реализации.

## 4. Что реально, что mock, что заглушка — сводная таблица

| Слой | Статус | Где живёт |
|---|---|---|
| UI-страницы (`/workday`, `/founder`, `/leads/[id]`, …) | **Реальный код**, рендерит mock-данные | `app/**/page.tsx` |
| Priority Score (1-100) | **Реальный детерминированный алгоритм** | `lib/scoring.ts` |
| Temperature (hot→expired) | **Реальный детерминированный алгоритм** | `lib/temperature.ts` |
| SLA (on_time/warning/overdue) | **Реальный детерминированный алгоритм** | `lib/sla.ts` |
| Money Priority Score | **Реальный детерминированный алгоритм** | `lib/moneyScore.ts` |
| Minimum Viable Lead / Source Confidence | **Реальная валидация** | `lib/validation.ts` |
| Исходные данные о лидах/офферах/скриптах | **Статичный mock**, в памяти процесса | `lib/mock.ts` |
| Реестр источников сигналов | **Статичный mock-реестр** (структура реальная, данные — нет) | `lib/sources.ts` |
| ~39 API-маршрутов | **Реальный код Next.js route handlers**, читающие mock-данные и явно помечающие `meta.status` | `app/api/**/route.ts` |
| База данных | **Не существует.** Черновик схемы, не применённый никуда | `db/schema.sql` |
| Внешние API (Google Places, 2GIS, Meta, WhatsApp, Telegram, goszakup, OpenAI/Anthropic) | **Не подключены.** Только переменные окружения-заготовки | `.env.example` |
| Авторизация / роли | **Не реализовано** | — (типы `users`/`user_role` есть только в `db/schema.sql`) |
| AI-генерация сообщений/скриптов/классификация | **Не реализовано.** Тексты — статичные mock-строки | `lib/mock.ts` (`whatsappMessage`, `followUp1d`, `scriptText`, …) |

## 5. Почему так и почему это нормально

Архитектурное решение этого раунда — **сначала спроектировать форму данных и
контракты API**, оставив реализацию интеграций как явно помеченный технический долг
(`pending_access`/`mock` в каждом ответе), а не пытаться имитировать "готовую"
интеграцию без реального доступа. Это значит:

- Подключение реального источника = написать fetch-логику внутри уже существующего
  `app/api/connectors/<source>/route.ts`, не меняя форму ответа.
- Подключение реальной БД = заменить `lib/mock.ts` на запросы к Supabase, не меняя
  типы из `lib/types.ts` (схема `db/schema.sql` уже спроектирована под них).
- Подключение AI = заменить статичные строки на вызовы LLM с тем же fallback-поведением.
